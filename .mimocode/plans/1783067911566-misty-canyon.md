# План: Подписки Lite и Pro

## Цель
Заменить текущую систему разового продвижения на подписочные планы Lite и Pro с переключателем ежемесячная/ежегодная оплата.

---

## Тарифы

| | **Lite** | **Pro** |
|---|---|---|
| Продвижение постов | ✅ | ✅ |
| Баннеры | 1 баннер | Без ограничений |
| Цена/мес | 1 500 ₽ | 4 000 ₽ |
| Цена/год | 15 000 ₽ (1 250 ₽/мес) | 40 000 ₽ (3 333 ₽/мес) |
| Экономия год | 17% | 17% |

---

## Шаг 1: Prisma Schema — модель `Subscription`

```prisma
model Subscription {
  id            String   @id @default(cuid())
  plan          String   // "lite" | "pro"
  period        String   // "monthly" | "yearly"
  status        String   @default("pending") // pending | active | expired | cancelled
  startDate     DateTime?
  endDate       DateTime?
  createdAt     DateTime @default(now())

  user          User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  userId        String

  @@index([userId])
  @@index([status])
}
```

Добавить в модель `User`:
```prisma
subscriptions  Subscription[]
```

## Шаг 2: SQL-миграция для Neon

```sql
CREATE TABLE IF NOT EXISTS "Subscription" (
    "id" TEXT NOT NULL,
    "plan" TEXT NOT NULL,
    "period" TEXT NOT NULL,
    "status" TEXT NOT NULL DEFAULT 'pending',
    "startDate" TIMESTAMP(3),
    "endDate" TIMESTAMP(3),
    "createdAt" TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,
    "userId" TEXT NOT NULL,
    CONSTRAINT "Subscription_pkey" PRIMARY KEY ("id")
);

CREATE INDEX IF NOT EXISTS "Subscription_userId_idx" ON "Subscription"("userId");
CREATE INDEX IF NOT EXISTS "Subscription_status_idx" ON "Subscription"("status");

ALTER TABLE "Subscription"
    ADD CONSTRAINT "Subscription_userId_fkey"
    FOREIGN KEY ("userId") REFERENCES "User"("id") ON DELETE CASCADE ON UPDATE CASCADE;
```

## Шаг 3: API подписок

### `GET /api/subscription` (авторизован)
- Возвращает текущую активную или ожидающую подписку пользователя
- Ответ: `{ subscription: { id, plan, period, status, startDate, endDate } | null }`

### `POST /api/subscription` (авторизован)
- Создаёт заявку на подписку
- Body: `{ plan: "lite"|"pro", period: "monthly"|"yearly" }`
- Проверка: нет ли уже активной/ожидающей подписки
- Статус: `pending`
- Ответ: `{ subscription }`

### `DELETE /api/subscription/[id]` (авторизован)
- Отмена подписки (только если `pending`)
- Проверка: принадлежит пользователю

---

## Шаг 4: API для проверки лимитов

### `GET /api/subscription/check` (авторизован)
- Проверяет, может ли пользователь создать продвижение/баннер
- Возвращает: `{ canPromote: boolean, canCreateBanner: boolean, bannerCount: number, maxBanners: number, plan: string | null }`
- Логика:
  - Нет подписки → `canPromote: false`
  - Lite → `canPromote: true`, `maxBanners: 1`
  - Pro → `canPromote: true`, `maxBanners: Infinity`

---

## Шаг 5: Обновление API продвижения

### `POST /api/promotion` — добавить проверку подписки
Перед созданием продвижения проверять:
- Есть ли активная подписка
- Доступно ли продвижение постов по тарифу

### `POST /api/promotion/banners` — добавить проверку лимита баннеров
Перед созданием баннера проверять:
- Есть ли активная подписка
- Не превышен ли лимит баннеров для тарифа (Lite = 1, Pro = ∞)

## Шаг 7: Админ-панель подписок

### `GET /api/admin/subscriptions` (ADMIN)
- Список всех подписок с фильтрацией по статусу

### `PATCH /api/admin/subscriptions` (ADMIN)
- Активация: статус `pending` → `active`, установка startDate/endDate
- Деактивация: `active` → `expired`

### Страница `/admin/subscriptions`
- Таблица: пользователь, план, период, статус, даты
- Кнопки «Активировать» / «Деактивировать»

## Верификация

1. `npx prisma generate` + `npx next build`
2. SQL миграция в Neon SQL Editor
3. Ручное тестирование:
   - Нет подписки → кнопка «Оформить» → заявка создана (pending)
   - Админ активирует → статус active
   - Пользователь видит активную подписку
   - Может создать продвижение/баннер
   - Lite: лимит 1 баннера → вторая попытка блокируется
   - Pro: без лимита
   - Переключатель месячный/годной меняет цену
   - Отмена pending-запроса работает
